в—Ҹ Zero type errors. Here's a precise walkthrough of every decision made:

  ---
  GraphQL strategy

  The query uses GitHub's search(type: ISSUE) API with merged:FROM..TO in the
  query string. The 120-day window is split into 14-day chunks вҖ” PostHog merges
  ~300вҖ“500 PRs/month, so each chunk stays comfortably below the 1000-result
  search cap. Within each chunk the loop paginates via pageInfo.endCursor until
  hasNextPage = false.

  __typename is fetched unconditionally so non-PR hits (issues) can be skipped
  without casting.

  Actor databaseId via inline fragments (scripts/ingest_github.ts:134-136,
  163-165): login lives on the Actor interface and works for all subtypes;
  databaseId only exists on User and Bot, so inline fragments are required.
  Mannequin and deleted-account ghost still provide login correctly.

  patch_excerpt = null вҖ” the GraphQL files connection exposes path + line counts
   but no diff text. The column is nullable and the enrichment prompt never uses
   it.

  enriched_at deliberately absent from both PrRow and PrReviewRow. On INSERT it
  defaults to NULL; on UPDATE (re-ingest conflict) Supabase only touches the
  columns in the payload, so the enrichment worker's stamp is preserved.

  State normalization
  - prs.state: GitHub GraphQL returns "MERGED" вҶ’ stored as "merged"
  (.toLowerCase())
  - pr_reviews.state: DISMISSED вҶ’ "COMMENTED" (DB CHECK only allows the three
  valid values); PENDING reviews are skipped outright (no submittedAt)
  - pr_changed_files.touch_type: COPIED вҶ’ "add" (closest semantic match within
  the CHECK constraint)

  n_comments vs n_inline_comments: n_inline_comments = comments.totalCount
  (GitHub's accurate total, not just the first 50 fetched); n_comments = 
  n_inline_comments + 1 when the top-level review body is non-empty.

  Upsert targets match the migration's unique constraints exactly:
  - prs вҶ’ onConflict: "id" (PK)
  - pr_changed_files вҶ’ onConflict: "pr_id,filename" (unique constraint
  pr_changed_files_pr_file_uq)
  - pr_reviews вҶ’ onConflict: "id" (PK)

  PRs are upserted before files/reviews on every page to satisfy the FK cascade.
