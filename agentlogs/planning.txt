  ---
  Key design decisions

  Plan.md names preserved exactly where they already existed (pr_scores,
  review_scores, pr_features). The task's requested names take precedence for
  the new split: engineer_weekly_velocity (plan called it
  engineer_weekly_scores), engineer_scores_90d, team_weekly_velocity,
  engineer_evidence, engineer_areas.

  pr_features vs pr_llm_features split ‚ÄĒ pr_features holds only deterministic
  structural signals (churn_ratio, cycle_time_hours, review_lag_hours,
  subsystem_list, counts). pr_llm_features holds everything Gemini produces plus
   the three auditability columns: model_name, prompt_version, input_hash
  (SHA-256 of the exact prompt user-turn). The cache index on (prompt_version, 
  input_hash) lets the enrichment worker skip re-calling Gemini when the prompt
  and input haven't changed.

  review_llm_features added ‚ÄĒ not in the minimum list but required by the
  pipeline described in plan.md; score-reviews.ts has nothing to read without
  it.

  pr_changed_files normalised out ‚ÄĒ prs no longer holds files_json; the GitHub
  files array maps to rows, enabling set-based subsystem extraction in SQL
  rather than JSON parsing in application code.

  team_weekly_velocity materialises the P25/P50/P75 band so /api/leaderboard
  reads a pre-computed row per week rather than calling percentile_cont() over
  engineer_weekly_velocity on every request.

  engineer_scores_90d ‚ÄĒ single row per engineer, replaced each rollup run. Gives
   the /api/leaderboard a constant-time read for the ranked table.
